## Terraform Provisioners

- It is used to execute scripts or commands on a local machine or a remote resource **after it has been created**.
- Provisioners are mainly used for post-deployment tasks such as: Installing software, Running shell scripts, Copying files, Configuring servers

---

### Why Use Provisioners?

- Provisioners help automate tasks after infrastructure is created.
- Examples: Install Apache or Nginx, Copy configuration files, Run setup scripts, Configure applications

---

### Provisioner Workflow

```text
Write Configuration
        │
        ▼
terraform apply
        │
        ▼
Resource Created
        │
        ▼
Provisioner Executes
        │
        ▼
Server Configured
```

---

### Types of Provisioners

Terraform provides two commonly used provisioners:

1. Local Provisioner (`local-exec`)
2. Remote Provisioner (`remote-exec`)

#### 1. local-exec

- The **local-exec** provisioner runs a command on the **machine where Terraform is executed**.
- Example:

```hcl
resource "aws_instance" "web" {

  ami           = "ami-xxxxxxxx"
  instance_type = "t2.micro"

  provisioner "local-exec" {
    command = "echo EC2 Instance Created"
  }
}
```

- When Terraform creates the EC2 instance, it prints:

```text
EC2 Instance Created
```

**Use Cases:**

* Send notifications
* Generate reports
* Update local files
* Trigger external scripts

---

#### 2. remote-exec

- The **remote-exec** provisioner runs commands **inside the remote server** after it is created.
- Example:

```hcl
resource "aws_instance" "web" {

  ami           = "ami-xxxxxxxx"
  instance_type = "t2.micro"

  connection {
    type        = "ssh"
    user        = "ec2-user"
    private_key = file("my-key.pem")
    host        = self.public_ip
  }

  provisioner "remote-exec" {
    inline = [
      "sudo yum update -y",
      "sudo yum install -y httpd",
      "sudo systemctl start httpd"
    ]
  }
}
```

- Terraform connects to the EC2 instance using SSH and executes the commands.

**Use Cases:**

* Install software
* Configure services
* Start applications
* Run setup scripts

---

### File Provisioner

- The **file** provisioner copies files from the local machine to a remote resource.
- Example:

```hcl
resource "aws_instance" "web" {

  provisioner "file" {
    source      = "index.html"
    destination = "/tmp/index.html"
  }
}
```

- A connection block is also required when using the `file` provisioner.

---

### Connection Block

- The **connection** block tells Terraform how to connect to the remote resource.
- Example:

```hcl
connection {
  type        = "ssh"
  user        = "ec2-user"
  private_key = file("my-key.pem")
  host        = self.public_ip
}
```

- Supported connection types: SSH (Linux), WinRM (Windows)

---

### Provisioner Lifecycle

- By default, provisioners run **after** a resource is created.
- They can also run when a resource is destroyed.
- Example:

```hcl
provisioner "local-exec" {
  when    = destroy
  command = "echo Resource Deleted"
}
```

---

### Real-World Example

Suppose Terraform creates an EC2 instance.
After creation, you want to:

* Install Apache
* Start the Apache service

Instead of logging into the server manually, Terraform executes the commands using `remote-exec`.

Workflow:

```text
Create EC2
      │
      ▼
Connect Using SSH
      │
      ▼
Install Apache
      │
      ▼
Start Service
```

---

### Best Practices

* Prefer **User Data** or **Cloud-Init** for server initialization.
* Use provisioners only when no better alternative exists.
* Avoid storing passwords or private keys directly in configuration files.
* Keep provisioner scripts simple and idempotent.
* Test commands before using them in production.

---

### Key Points

* Provisioners execute commands after a resource is created or before it is destroyed.
* `local-exec` runs commands on the local machine.
* `remote-exec` runs commands on the remote resource.
* `file` copies files from the local machine to the remote resource.
* A `connection` block is required for `remote-exec` and `file`.
* HashiCorp recommends using provisioners only when other solutions are not suitable.
