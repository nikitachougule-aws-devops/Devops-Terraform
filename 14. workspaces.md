## Terraform Workspaces

- It allows you to manage **multiple environments** using the same Terraform configuration.
- Each workspace has its **own separate state file**, so resources in one workspace do not affect another.
- Common environments: Development (dev), Testing (test), Staging (stage), Production (prod)

---

## Why Use Workspaces?

Without workspaces:

* Separate folders are needed for each environment.
* Code is duplicated.
* Maintenance becomes difficult.

With workspaces:

* One configuration can manage multiple environments.
* Each environment has its own state.
* Easier to manage infrastructure.

---

## Workspace Concept

```text
Same Terraform Code
         │
         ▼
 ┌────────┬────────┬
 │        │        │
 ▼        ▼        ▼
dev     test     prod
 │        │        │
 ▼        ▼        ▼
State    State    State
File     File     File
```

---

## Default Workspace

- When you initialize a Terraform project, Terraform automatically creates the **default** workspace.
- Check the current workspace:

```bash
terraform workspace show
```

Output:

```text
default
```

---

## Common Workspace Commands

### List Workspaces

```bash
terraform workspace list
```

Example output:

```text
* default
  dev
  test
  prod
```

The `*` indicates the current workspace.

---

### Create a Workspace

```bash
terraform workspace new dev
```

Creates a new workspace named **dev**.

---

### Switch Workspace

```bash
terraform workspace select dev
```

Switches to the **dev** workspace.

---

### Show Current Workspace

```bash
terraform workspace show
```

Displays the active workspace.

---

### Delete a Workspace

```bash
terraform workspace delete dev
```

Deletes the **dev** workspace.

> You cannot delete the workspace you are currently using. Switch to another workspace first.

---

## Using Workspace Name

You can access the current workspace using:

```hcl
terraform.workspace
```

Example:

```hcl
resource "aws_instance" "web" {
  tags = {
    Environment = terraform.workspace
  }
}
```

If the current workspace is **dev**, the EC2 instance will have:

```text
Environment = dev
```

## Workspace Workflow

```text
terraform init
        │
        ▼
terraform workspace new dev
        │
        ▼
terraform workspace select dev
        │
        ▼
terraform apply
        │
        ▼
Resources Created for dev
```

## Real-World Example

Suppose your application has three environments:

* Development
* Testing
* Production

Create workspaces:

```bash
terraform workspace new dev
terraform workspace new test
terraform workspace new prod
```

Switch to the development environment:

```bash
terraform workspace select dev
terraform apply
```

Switch to production:

```bash
terraform workspace select prod
terraform apply
```

Terraform creates separate resources for each workspace because each workspace has its own state file.

---

## Best Practices

* Use workspaces for environments that share the same infrastructure design.
* Use meaningful workspace names such as `dev`, `test`, and `prod`.
* Always verify the current workspace before running `terraform apply`.
* Use separate backends or directories for large production environments when required.
* Avoid making changes in the wrong workspace.

---

## Key Points

* A **workspace** allows multiple environments to use the same Terraform configuration.
* Each workspace has its own **state file**.
* The default workspace is **default**.
* Use `terraform workspace new` to create a workspace.
* Use `terraform workspace select` to switch workspaces.
* Use `terraform workspace show` to display the current workspace.
* Access the current workspace name with `terraform.workspace`.
