## Backend and Remote State

- It is the place where Terraform stores its **state file**.
- A backend allows Terraform to store the state **locally** or **remotely**.
- By default, Terraform stores the state locally in:

```text
terraform.tfstate
```
---

### Why Use a Backend?

A backend helps Terraform:

* Store the state file.
* Share the state with team members.
* Lock the state to prevent simultaneous changes.
* Secure the state file.
* Improve collaboration.

---

### Types of Backends

Terraform supports two types of backends:

 Local Backend

- It stores the state file on your local computer.
- Example:

```text
project/
│
├── main.tf
├── variables.tf
└── terraform.tfstate
```

Advantages:

* Simple to use.
* Good for learning.
* No additional setup.

Disadvantages:

* Not suitable for teams.
* No state locking.
* Risk of losing the state file.

---

2. Remote Backend

- It stores the state file on a remote storage service.

Common remote backends:

* AWS S3
* Azure Storage
* Google Cloud Storage
* Terraform Cloud

Advantages:

* Shared by the team.
* Supports state locking (depending on the backend).
* More secure.
* Suitable for production environments.

---

### Backend Configuration Example

Example using an AWS S3 backend:

```hcl
terraform {
  backend "s3" {
    bucket = "my-terraform-state"
    key    = "dev/terraform.tfstate"
    region = "us-east-1"
  }
}
```

Run:

```bash
terraform init
```

Terraform initializes the backend and connects to the remote state.

---

## Remote State

- It means storing the Terraform state file in a remote backend instead of on your local machine.
- Example:

```text
Developer
     │
     ▼
Terraform
     │
     ▼
Remote Backend (S3 / Terraform Cloud)
     │
     ▼
terraform.tfstate
```

- All team members use the same shared state file.

---

## Backend Workflow

```text
Write Configuration
        │
        ▼
Configure Backend
        │
        ▼
terraform init
        │
        ▼
Backend Initialized
        │
        ▼
terraform apply
        │
        ▼
State Stored Remotely
```

---

## Local State vs Remote State

| Local State              | Remote State                               |
| ------------------------ | ------------------------------------------ |
| Stored on local computer | Stored on a remote backend                 |
| Best for learning        | Best for teams and production              |
| Not shared               | Shared by the team                         |
| No state locking         | Supports state locking (backend dependent) |
| Risk of loss             | More reliable and secure                   |

## Real-World Example

Suppose three developers are working on the same Terraform project.

If each developer keeps a local state file:

* Different state files may become inconsistent.
* One developer may overwrite another's changes.

Using an **S3 backend**, everyone shares the same state file.

Workflow:

1. Configure the S3 backend.
2. Run `terraform init`.
3. Terraform stores the state in S3.
4. Every developer works with the same remote state.

---

## Best Practices

* Use a remote backend for team and production environments.
* Never edit the state file manually.
* Protect the backend with proper access controls.
* Enable versioning on the storage service (for example, S3 Versioning).
* Do not commit `terraform.tfstate` to Git.

---

## Key Points

* A **backend** determines where Terraform stores its state file.
* The default backend is **local**.
* A **remote backend** stores the state in a shared location.
* Remote state improves collaboration and security.
* Configure the backend in the `terraform` block.
* Run `terraform init` after adding or changing a backend configuration.
