## Terraform Lifecycle

- It describes how Terraform manages a resource throughout its life, from creation to deletion.
- Terraform automatically determines whether a resource should be:

1. Created
2. Updated
3. Replaced
4. Destroyed

You can also customize this behavior using the **`lifecycle`** block.

---

## Resource Lifecycle

```text
Write Configuration
        │
        ▼
terraform plan
        │
        ▼
Create Resource
        │
        ▼
Update Resource
        │
        ▼
Replace Resource (if needed)
        │
        ▼
Destroy Resource
```

---

## Lifecycle Operations

### 1. Create

If a resource does not exist, Terraform creates it.

```bash
terraform apply
```

Example:

```hcl
resource "aws_instance" "web" {
  ami           = "ami-xxxxxxxx"
  instance_type = "t2.micro"
}
```

---

### 2. Update

If you modify the configuration, Terraform updates the existing resource whenever possible.

Example:

Before:

```hcl
instance_type = "t2.micro"
```

After:

```hcl
instance_type = "t3.micro"
```

Run:

```bash
terraform apply
```

Terraform updates only the changed attribute.

---

### 3. Replace

Some changes cannot be updated on an existing resource.

Terraform deletes the old resource and creates a new one.

Example:

```text
Old EC2 Instance
        │
   Delete
        │
        ▼
New EC2 Instance
```

Terraform shows this in the execution plan before making changes.

---

### 4. Destroy

Delete all resources managed by Terraform.

```bash
terraform destroy
```

Terraform removes all managed infrastructure.

---

## Lifecycle Meta-Argument

The **`lifecycle`** block controls how Terraform manages a resource.

Syntax:

```hcl
resource "<resource_type>" "<resource_name>" {

  lifecycle {
    ...
  }
}
```

---

## Common Lifecycle Rules

Terraform provides four commonly used lifecycle rules:

* `create_before_destroy`
* `prevent_destroy`
* `ignore_changes`
* `replace_triggered_by`

---

### 1. create_before_destroy

Creates the new resource before deleting the old one.

```hcl
resource "aws_instance" "web" {
  ami           = "ami-xxxxxxxx"
  instance_type = "t2.micro"

  lifecycle {
    create_before_destroy = true
  }
}
```

**Use Case:**

Avoid downtime while replacing resources.

---

### 2. prevent_destroy

Prevents accidental deletion of a resource.

```hcl
resource "aws_db_instance" "db" {

  lifecycle {
    prevent_destroy = true
  }
}
```

If someone runs:

```bash
terraform destroy
```

Terraform stops the operation and displays an error.

**Use Case:**

* Production databases
* Critical infrastructure

---

### 3. ignore_changes

Ignores changes made to selected resource attributes.

```hcl
resource "aws_instance" "web" {

  lifecycle {
    ignore_changes = [
      tags
    ]
  }
}
```

Terraform ignores changes to the `tags` attribute.

**Use Case:**

When another tool updates tags or metadata.

---

### 4. replace_triggered_by

Replaces a resource when another resource changes.

```hcl
resource "aws_instance" "web" {

  lifecycle {
    replace_triggered_by = [
      aws_security_group.web
    ]
  }
}
```

**Use Case:**

Replace an EC2 instance whenever its security group changes.

## Lifecycle Workflow

```text
Write Configuration
        │
        ▼
terraform plan
        │
        ▼
Terraform Compares
Configuration + State
        │
        ▼
Create / Update / Replace / Destroy
        │
        ▼
terraform apply
```

---

## Real-World Example

Suppose you have a production database.

You want to prevent accidental deletion.

```hcl
resource "aws_db_instance" "db" {

  lifecycle {
    prevent_destroy = true
  }
}
```

Now, even if someone runs:

```bash
terraform destroy
```

Terraform blocks the deletion.

---

## Best Practices

* Use `create_before_destroy` to minimize downtime.
* Use `prevent_destroy` for critical resources.
* Use `ignore_changes` only when necessary.
* Review `terraform plan` before applying changes.
* Avoid unnecessary resource replacements.

---

## Key Points

* Terraform automatically manages the lifecycle of resources.
* Resources can be **created**, **updated**, **replaced**, or **destroyed**.
* The `lifecycle` block customizes this behavior.
* `create_before_destroy` creates new resources before deleting old ones.
* `prevent_destroy` protects critical resources.
* `ignore_changes` ignores selected attribute changes.
* `replace_triggered_by` forces resource replacement when another resource changes.
* Always review `terraform plan` before running `terraform apply`.
