## 1. What is Terraform?

**Answer:**

Terraform is an **Infrastructure as Code (IaC)** tool developed by HashiCorp. It is used to create, update, and manage infrastructure using configuration files.

---

## 2. What is Infrastructure as Code (IaC)?

**Answer:**

Infrastructure as Code (IaC) is the practice of managing infrastructure using code instead of manual configuration.

---

## 3. What are the advantages of Terraform?

**Answer:**

* Open source
* Multi-cloud support
* Infrastructure as Code
* Version control support
* Reusable modules
* Automatic dependency management

---

## 4. What is a Provider?

**Answer:**

A **Provider** is a plugin that allows Terraform to communicate with cloud platforms like AWS, Azure, and GCP.

Example:

```hcl
provider "aws" {
  region = "us-east-1"
}
```

---

## 5. What is a Resource?

**Answer:**

A **Resource** is an infrastructure component managed by Terraform, such as an EC2 instance, S3 bucket, or VPC.

---

## 6. What is the Terraform State File?

**Answer:**

The **state file (`terraform.tfstate`)** stores information about the infrastructure managed by Terraform.

Terraform uses it to track resources and detect changes.

---

## 7. What is the difference between `terraform plan` and `terraform apply`?

**Answer:**

* `terraform plan` previews the changes.
* `terraform apply` creates or updates the infrastructure.

---

## 8. What is `terraform init`?

**Answer:**

`terraform init` initializes a Terraform project by downloading providers and preparing the working directory.

---

## 9. What is a Module?

**Answer:**

A **module** is a reusable collection of Terraform configuration files.

It helps organize code and avoid duplication.

---

## 10. What are Variables?

**Answer:**

Variables are input values that make Terraform configurations reusable.

Example:

```hcl
variable "instance_type" {
  default = "t2.micro"
}
```

---

## 11. What are Outputs?

**Answer:**

Outputs display useful information after Terraform creates resources.

Example:

* Instance ID
* Public IP
* VPC ID

---

## 12. What is a Backend?

**Answer:**

A **backend** defines where Terraform stores the state file.

Examples:

* Local
* AWS S3
* Azure Storage
* Terraform Cloud

---

## 13. What is Remote State?

**Answer:**

Remote state stores the Terraform state file in a shared remote backend, making collaboration easier.

---

## 14. What is a Workspace?

**Answer:**

A workspace allows multiple environments (such as dev, test, and prod) to use the same Terraform configuration with separate state files.

---

## 15. What is the Lifecycle Block?

**Answer:**

The `lifecycle` block controls how Terraform manages resources.

Common options:

* `create_before_destroy`
* `prevent_destroy`
* `ignore_changes`
* `replace_triggered_by`

---

## 16. What is a Provisioner?

**Answer:**

A provisioner runs commands or scripts after a resource is created or before it is destroyed.

Examples:

* `local-exec`
* `remote-exec`

---

## 17. What is the difference between Local State and Remote State?

**Answer:**

| Local State             | Remote State               |
| ----------------------- | -------------------------- |
| Stored on local machine | Stored on a remote backend |
| Best for learning       | Best for teams             |
| Not shared              | Shared across users        |

---

## 18. What is the difference between `terraform destroy` and deleting resources manually?

**Answer:**

* `terraform destroy` removes resources and updates the state file.
* Manual deletion removes resources from the cloud but leaves the state file unchanged, causing state drift.

---

## 19. What is State Drift?

**Answer:**

State drift occurs when infrastructure is changed manually outside Terraform, causing the actual infrastructure to differ from the state file.

---

## 20. What happens when you run `terraform plan`?

**Answer:**

Terraform compares:

* Configuration files
* State file
* Actual infrastructure

Then it shows what will be created, updated, replaced, or deleted.

---

## 21. What is HCL?

**Answer:**

HCL (HashiCorp Configuration Language) is the language used to write Terraform configuration files.

---

## 22. What is `terraform fmt`?

**Answer:**

`terraform fmt` formats Terraform configuration files according to standard formatting rules.

---

## 23. What is `terraform validate`?

**Answer:**

`terraform validate` checks Terraform configuration files for syntax and configuration errors.

---

## 24. What is `terraform import`?

**Answer:**

`terraform import` adds an existing cloud resource to the Terraform state so Terraform can manage it.

---

## 25. What is the Terraform Workflow?

**Answer:**

The standard workflow is:

```text
Write Code
    │
    ▼
terraform init
    │
    ▼
terraform validate
    │
    ▼
terraform plan
    │
    ▼
terraform apply
    │
    ▼
terraform destroy (optional)
```

# Scenario-Based Interview Questions

## 1. You accidentally deleted the state file. What will you do?

**Answer:**

* Restore it from backup.
* If using a remote backend, reinitialize using `terraform init`.

---

## 2. `terraform apply` fails. What will you do?

**Answer:**

* Read the error message.
* Fix the issue (configuration, permissions, or credentials).
* Run `terraform plan` and then `terraform apply` again.

---

## 3. A resource already exists, but Terraform wants to create it. What will you do?

**Answer:**

Import the existing resource.

```bash
terraform import <resource_type>.<resource_name> <resource_id>
```

---

## 4. Your state file is locked. What will you do?

**Answer:**

Wait for the current operation to finish.

If the lock is stuck:

```bash
terraform force-unlock <LOCK_ID>
```

---

## 5. Someone manually changed a resource in AWS. What will you do?

**Answer:**

Run:

```bash
terraform plan
```

Review the differences and apply the required changes.
