## Terraform State File

- It is a file that stores information about the infrastructure managed by Terraform.
- Terraform uses the state file to keep track of the resources it has created.
- By default, the state file is named:

```text
terraform.tfstate
```

---

## Why is the State File Important?

The state file helps Terraform:

* Track existing resources.
* Compare the current infrastructure with the configuration.
* Detect changes.
* Update only the required resources.
* Avoid creating duplicate resources.

Without the state file, Terraform would not know which resources it already manages.

---

## How the State File Works

```text
Terraform Configuration
         │
         ▼
terraform apply
         │
         ▼
Cloud Resources Created
         │
         ▼
terraform.tfstate Updated
```

Whenever you run `terraform apply`, Terraform updates the state file.

---

## Example

Suppose you create an EC2 instance.

Configuration:

```hcl
resource "aws_instance" "web" {
  ami           = "ami-xxxxxxxx"
  instance_type = "t2.micro"
}
```

After running:

```bash
terraform apply
```

Terraform:

1. Creates the EC2 instance.
2. Stores its details in `terraform.tfstate`.

The state file contains information such as:

* Resource ID
* Resource type
* Attributes
* Dependencies

```text

        ▼
terraform plan

        ▼

        │
        ▼
Resources Created
        │
        ▼
terraform.tfstate Updated
```

## Local State

By default, Terraform stores the state file locally.

Example:

```text
project/
│
├── main.tf
├── variables.tf
├── outputs.tf
└── terraform.tfstate
```

Local state is suitable for learning and small projects.

---

## Remote State

In team environments, the state file is usually stored remotely.

Common remote backends include:

* AWS S3
* Azure Storage
* Google Cloud Storage
* Terraform Cloud

Benefits:

* Shared by the team.
* Prevents multiple users from overwriting the state.
* More secure than storing it locally.
* Supports state locking (depending on the backend).

---

## View the State

Display the current state:

```bash
terraform show
```

List all resources in the state:

```bash
terraform state list
```

View a specific resource:

```bash
terraform state show aws_instance.web
```

---

## Common State Commands

| Command                           | Purpose                                                                     |
| --------------------------------- | --------------------------------------------------------------------------- |
| `terraform show`                  | Display the current state                                                   |
| `terraform state list`            | List all resources                                                          |
| `terraform state show <resource>` | Display resource details                                                    |
| `terraform state rm <resource>`   | Remove a resource from the state file (does not delete the actual resource) |
| `terraform import`                | Import an existing resource into the state                                  |

---

## Important Notes

* **Do not edit `terraform.tfstate` manually.**
* The state file may contain **sensitive information**, such as resource IDs or configuration details.
* Add the state file to `.gitignore` unless you are using an approved remote backend.
* Use a remote backend for production environments.

---

## Real-World Example

Suppose you create an EC2 instance using Terraform.

**First run:**

```bash
terraform apply
```

Terraform:

* Creates the EC2 instance.
* Saves its details in `terraform.tfstate`.

Later, you change:

```hcl
instance_type = "t3.micro"
```

Run:

```bash
terraform apply
```

Terraform compares the configuration with the state file and updates only the EC2 instance instead of creating a new one.

## Key Points

* The default Terraform state file is **`terraform.tfstate`**.
* It stores information about infrastructure managed by Terraform.
* Terraform uses the state file to track, compare, and update resources.
* Do not edit the state file manually.
* Use a remote backend for team and production environments.
* Use `terraform state` commands to inspect and manage the state safely.
