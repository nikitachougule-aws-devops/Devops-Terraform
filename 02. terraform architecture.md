## Terraform Architecture

- Explains how Terraform works internally to create and manage infrastructure.
- Terraform reads your configuration files, creates an execution plan, communicates with cloud providers, and keeps track of infrastructure using a state file.

---

## Terraform Architecture Diagram

```text
                 Terraform Configuration
                       (*.tf files)
                             │
                             ▼
                    Terraform CLI
            (init, plan, apply, destroy)
                             │
                             ▼
                     Terraform Core
          (Reads configuration & creates plan)
                             │
          ┌──────────────────┴──────────────────┐
          ▼                                     ▼
     State File                          Provider Plugins
 (terraform.tfstate)                          │
                                              ▼
                                   Cloud Provider APIs
                                (AWS, Azure, GCP, etc.)
                                              │
                                              ▼
                                     Cloud Resources
```

## Components of Terraform Architecture

Terraform architecture consists of the following components:

### 1. Configuration Files

Terraform configurations are written in **HCL (HashiCorp Configuration Language)**.

Example:

```hcl
provider "aws" {
  region = "us-east-1"
}

resource "aws_instance" "web" {
  ami           = "ami-xxxxxxxx"
  instance_type = "t2.micro"
}
```

These files describe the infrastructure you want Terraform to create.

---

### 2. Terraform CLI

The **Terraform CLI (Command Line Interface)** is used to run Terraform commands.

Common commands:

```bash
terraform init
terraform plan
terraform apply
terraform destroy
```

The CLI processes your configuration and sends it to Terraform Core.

---

### 3. Terraform Core

Terraform Core is the main engine of Terraform.

It performs the following tasks:

* Reads configuration files.
* Creates an execution plan.
* Determines resource dependencies.
* Communicates with providers.
* Updates the state file.

Terraform Core does **not** directly create cloud resources.

---

### 4. Providers

A **Provider** is a plugin that allows Terraform to communicate with a cloud platform or service.

Examples:

* AWS Provider
* Azure Provider
* Google Cloud Provider
* Kubernetes Provider

Example:

```hcl
provider "aws" {
  region = "us-east-1"
}
```

---

### 5. Cloud APIs

Providers communicate with the cloud provider through APIs.

Example:

```text
Terraform
     │
AWS Provider
     │
AWS API
     │
EC2 Instance Created
```

Terraform never creates resources directly. It sends API requests through the provider.

---

### 6. State File

Terraform stores information about the infrastructure in a **state file**.

Default state file:

```text
terraform.tfstate
```

The state file helps Terraform:

* Track existing resources.
* Detect changes.
* Update only modified resources.
* Avoid creating duplicate resources.

## Terraform Working Process

```text
Write Configuration
        │
        ▼
terraform init
        │
        ▼
Download Providers
        │
        ▼
terraform plan
        │
        ▼
Review Execution Plan
        │
        ▼
terraform apply
        │
        ▼
Provider Calls Cloud API
        │
        ▼
Resources Created
        │
        ▼
State File Updated
```

---

## Example

Suppose you want to create an AWS EC2 instance.

1. Write the configuration in a `.tf` file.
2. Run `terraform init` to download the AWS provider.
3. Run `terraform plan` to preview the changes.
4. Run `terraform apply` to create the EC2 instance.
5. Terraform stores the resource information in `terraform.tfstate`.

---

## Key Points

* Terraform uses **HCL** configuration files.
* **Terraform CLI** runs Terraform commands.
* **Terraform Core** reads configurations and creates an execution plan.
* **Providers** communicate with cloud platforms.
* **Cloud APIs** create, update, and delete resources.
* The **State File** keeps track of the infrastructure managed by Terraform.
